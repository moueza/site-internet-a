Type de zone Un serveur DNS peut être configuré selon les modes suivant: Master: Serveur DNS de type primaire, il a autorité sur la zone et permet de déclarer de nouvelles zones. Slave: Serveur DNS de type secondaire, c'est une copie d'un serveur primaire, il contient les même informations que le serveur primaire et se met à jour si les informations contenues dans le serveur DNS primaire changent. Un serveur DNS secondaire est, le plus souvent, mis en place pour alléger la charge de travail du serveur DNS primaire, mais aussi pour la tolérance de panne, le serveur secondaire prenant le relais si le serveur primaire venait à tomber en panne. Stub: Une zone dite «stub» marche, en un sens, comme un serveur de type «slave», sauf qu'une zone stub ne contient que les enregistrements de type NS. Une zone stub n'est pas une zone à proprement parler, elle remplie plutôt le rôle d'un «index» auquel la ou les zones associées vont se référer. Ce type de configuration est surtout utilisée lorsque plusieurs zones aux noms différents ont besoin d'avoir accès à la même base de données. Forward: Un serveur de type «forward» ne contient aucune données pour la zone en question, il se contente de rediriger les requêtes vers d'autres serveurs DNS donnés. Un serveur de type forward peut se révéler utile si le serveur maitre a à supporter beaucoup de trafic. IV. Délégation de zone Une délégation de zone permet d'avoir un serveur qui s'occupe de manière exclusive d'un sous-domaine donné. Le serveur maitre se contente de rediriger les requêtes destinées au sousdomaine vers le serveur DNS en charge de celui-ci. Une délégation de zone peut se révéler utile pour alléger la charge de travail du serveur primaire, ou si le sous-domaine nécessite une configuration particulière. Pour ajouter un enregistrement au sous-domaine, il n'est donc plus nécessaire de l'ajouter sur le DNS de la zone principale, il suffit de l'ajouter sur le DNS du sous-domaine. Ceci permet donc une meilleur flexibilité et une meilleur visibilité, on évite de devoir modifier la configuration du DNS parent. Dans ce cadre, une zone dite «Glue» correspond à l'ensemble des enregistrements NS des serveur DNS des zones déléguées. Il est ainsi possible pour le serveur primaire de savoir quel serveur interroger pour chaque requêtes que le serveur primaire ne gère pas. Si on se réfère aux définitions données plus haut, on se rend compte qu'il n'est pas possible de faire de la délégation de zone en utilisant une zone stub, puisque celle-ci ne contiendra que les enregistrements de type NS, et pas les autres enregistrements, elle ne peut donc pas assurer le rôle de serveur DNS. 4 V. Suffixes DNS Un suffixe DNS est un nom de zone ajouté après le nom d'une machine. L'utilisation d'un suffixe DNS permet, entre autre, d accéder directement à une machine du domaine en ne renseignant que son nom. Une machine peut avoir plusieurs suffixes DNS, puisqu'une machine peut posséder plusieurs interface réseau, mais une interface réseau peut aussi avoir plusieurs suffixes DNS par ordre de priorité. Pour définir un suffixe DNS (sous Windows) il faut: Dans les paramètres du nom de l'ordinateur. (clic droit sur «poste de travail» propriétés onglet «identification réseau» propriétés «Autres» et ajouter un suffixe DNS principal pour la machine) Dans les propriétés TCP/IP. («avancé» onglet DNS «ajouter ces suffixes DNS (dans l'ordre)» ) Il est possible de cette manière de renseigner plusieurs suffixes DNS que l'os essaiera l'un après l'autre en cas d'échec, dans l'ordre où ces suffixes ont été placé par l'utilisateur VI. VIII. Dialogue entre maitre et esclave On se propose maintenant d'étudier les différents dialogues entre Serveur3 et Serveur4, respectivement maitre et esclave, et plus particulièrement les dialogues se déroulant lorsqu'on change un enregistrement dans Serveur3. Premier cas: Le serveur maitre envoie un paquet «DNS notify», notifiant à ses serveurs esclaves que sa base de données a changée. Ces derniers se mettent donc à jour. Deuxième cas: Le serveur esclave envoie une requête au serveur maitre, lui demandant quels ont été les changements depuis la dernière mise à jour. Le serveur maitre lui envoie donc les informations demandées, s'il y a lieu. Serveur3 ouvre une connexion de type TCP (identifiée, entre autre, par les paquets SYN et FIN) et lui envoie les informations demandées. Ceci fait, la connexion est close, et les serveurs maitre et esclave possèdent maintenant les même enregistrements.
